## 2. 迁移目标架构

```
┌─────────────────────────────────────────────────────────┐
│                    CloudBase 环境                        │
│                                                         │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  Node.js     │  │  Node.js     │  │  Node.js     │  │
│  │  云函数组     │  │  云函数组     │  │  云函数组     │  │
│  │  (WX 商城端) │  │  (WX 管理端) │  │  (Admin 面向) │  │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘  │
│         │                 │                  │          │
│         ▼                 ▼                  ▼          │
│  ┌─────────────────────────────────────────────────┐   │
│  │              CloudBase MySQL 8.0                 │   │
│  │         (表结构直迁，保留原有设计)                 │   │
│  └─────────────────────────────────────────────────┘   │
│                                                         │
│  ┌──────────────┐  ┌──────────────────────────────┐    │
│  │  云存储 (COS) │  │  静态托管 (Vue 管理后台)      │    │
│  └──────────────┘  └──────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘

调用关系：
  微信小程序商城端 ──wx.cloud.callFunction()──► 云函数组 (WX 商城) ──► MySQL
  微信小程序管理端 ──wx.cloud.callFunction()──► 云函数组 (WX 管理) ──► MySQL
  管理后台 Vue ──JS SDK callFunction()──► 云函数组 (Admin) ──► MySQL
  H5 移动端 ──暂不迁移（保留 REST API 或后续接入）
  （管理后台通过 CloudBase 静态托管默认域名访问，低频使用，无需自定义域名）
```

## 3. 技术选型决策

### 3.1 后端：Node.js 云函数（非 CloudRun）

| 决策 | 理由 |
|------|------|
| 语言选 Node.js 18 | 云函数一等公民，生态最好，冷启动优于 Java |
| 不用 CloudRun | 用户明确不希望走云托管 |
| 不用 Java 云函数 | JVM 冷启动 5-10 秒，不可接受 |

### 3.2 数据库：CloudBase MySQL（非 NoSQL）

| 决策 | 理由 |
|------|------|
| 保持 MySQL | 47 个实体类的电商系统，关联查询复杂，NoSQL 不适合 |
| NoSQL 100 集合上限 | 当前 40+ 张表 + 关联，上限吃紧 |
| 表结构直迁 | 避免重写数据层，降低迁移风险 |
| 可行性高 | CloudBase MySQL 基于 TDSQL-C，兼容 MySQL 8.0 |

### 3.3 认证方案

| 端 | 当前 | 迁移后 |
|---|------|--------|
| 小程序 | JWT（自签） | CloudBase 自动注入 OPENID，云函数端 `cloud.getWXContext()` 获取 |
| 管理后台 | Shiro + Session | CloudBase Auth v2（用户名/密码登录）+ JS SDK |

### 3.4 存储：CloudBase 云存储

| 当前 | 迁移后 |
|------|--------|
| 阿里云 OSS（`clothing-mall-2026` bucket，深圳区域） | CloudBase 云存储（底层 COS） |

### 3.5 管理后台部署

| 当前 | 迁移后 |
|------|--------|
| Nginx 反代 + Docker | CloudBase 静态托管，Vue SPA 构建后上传 |

### 3.6 H5 移动端

| 当前 | 迁移后 |
|------|--------|
| Vue 2 + Vant，部署在 Nginx | **暂不迁移**，保持 REST API 调用方式；后续可接入云函数 |

- 减少冷启动概率

- 单函数代码包更紧凑

### 4.5 定时任务云函数（4 个函数）

> 项目当前有 6 个定时任务，按职责合并为 4 个云函数。

| 函数名 | 职责 | Cron | 对应现有任务 |
|--------|------|------|-------------|
| `task-order` | 超时未付自动取消 + 超期自动确认 | `0 */5 * * * *`（每 5 分钟检查取消）；`0 0 3 * * *`（每天 3:00 确认收货） | OrderUnpaidTask, OrderJob.checkOrderUnconfirm |
| `task-coupon` | 优惠券过期检查 + 生日券发放 | `0 0 * * * *`（每小时检查过期）；`0 30 0 * * *`（每天 0:30 生日券） | CouponJob.checkCouponExpired, BirthdayCouponJob |
| `task-push` | 定时推送（企微消息） | `0 * * * * *`（每分钟检查） | ScheduledPushTask |
| `task-backup` | 数据库备份 | `0 0 5 * * *`（每天 5:00） | DbJob.backup |

**注意**：当前 OrderUnpaidTask 使用 Java DelayQueue 实现，云函数中需改为定时轮询方案（每 5 分钟扫描 `order_status=101 AND add_time < NOW()-30min`）。

### 4.6 共享层（Layer）

将公共依赖抽取为共享层，减少各函数代码包体积：

| 层名 | 内容 |
|------|------|
| `layer-base` | mysql2、数据库连接池、工具函数、响应格式化 |
| `layer-auth` | 认证鉴权中间件（OPENID 解析、Admin 权限校验、管理端管理员校验） |
| `layer-wechat` | 企业微信 API 封装、火山引擎 AI 调用封装 |

### 4.7 单个云函数内部结构（示例）

```
wx-order/
├── index.js          # 入口，路由分发
├── service/
│   ├── order.js      # 订单业务逻辑
│   └── aftersale.js  # 售后逻辑
└── package.json
```

**入口文件示例**：
```javascript
const { getOrderRoutes } = require('./service/order')
const { getAftersaleRoutes } = require('./service/aftersale')

exports.main = async (event, context) => {
  const { action, data } = event
  const routes = {
    ...getOrderRoutes(),
    ...getAftersaleRoutes(),
  }

  const handler = routes[action]
  if (!handler) {
    return { errno: 404, errmsg: `Unknown action: ${action}` }
  }
  return handler(data, context)
}
```

## 5. 数据库迁移方案

### 5.1 迁移策略：表结构直迁

| 步骤 | 操作 | 工具 |
|------|------|------|
| 1 | 阿里云 MySQL 导出 | `mysqldump -u root -p clothing_mall > backup.sql` |
| 2 | CloudBase 创建 MySQL 实例 | MCP: `manageSqlDatabase(action=provisionMySQL)` |
| 3 | 导入 SQL 到 CloudBase MySQL | DMC 平台 或 MCP: `manageSqlDatabase(action=initializeSchema)` |
| 4 | 验证表结构和数据完整性 | MCP: `querySqlDatabase(action=runQuery)` |

### 5.2 表结构调整

- 移除 Shiro 相关表（`litemall_admin` 的 `session_id` 等字段改用 CloudBase Auth）
- JWT 相关字段不再需要
- 新增 CloudBase 相关字段（如 `openid` 映射到 `litemall_user`）
- OSS 存储 URL 需批量更新为云存储 fileID（OSS → COS 迁移）
- 已删除的表无需迁移（flash_sale、full_reduction、member_level 等，V1.0.23 已清理）

### 5.3 云函数连接 MySQL

```javascript
// layer-base 中提供共享连接池
const mysql = require('mysql2/promise')

let pool = null
function getPool() {
  if (!pool) {
    pool = mysql.createPool({
      host: process.env.MYSQL_HOST,
      port: parseInt(process.env.MYSQL_PORT || '3306'),
      user: process.env.MYSQL_USER,
      password: process.env.MYSQL_PASSWORD,
      database: process.env.MYSQL_DATABASE,
      connectionLimit: 5,
      waitForConnections: true,
    })
  }
  return pool
}
```

## 7. 前端改造方案

### 7.1 小程序改造

**改动范围：中**。核心变化：

1. `app.js` 增加云开发初始化（`wx.cloud.init()`）
2. 封装 `cloudCall` 工具函数替代 `util/util.js` 中的 `wx.request`
3. 登录流程改为 CloudBase 身份体系（静默登录改为 OPENID 自动获取）
4. 文件上传改为 `wx.cloud.uploadFile`
5. **管理端页面需适配**（14 个页面：tabOrder、tabShelf、goodsEdit、orderDetail 等）

```javascript
// 改造前 (utils/util.js)
function request(url, data) {
  return new Promise((resolve, reject) => {
    wx.request({
      url: WxApiRoot + url,
      method: 'POST',
      data: data,
      header: { 'X-Litemall-Token': token },
      success: res => resolve(res.data),
      fail: reject
    })
  })
}

// 改造后
function cloudCall(funcName, action, data) {
  return wx.cloud.callFunction({
    name: funcName,
    data: { action, data }
  }).then(res => res.result)
}
```

**管理端页面特别说明**：
- `tabOrder` 调用 `wx-manager-order`，`tabShelf` 调用 `wx-manager-shelf`
- `sceneManage/sceneEdit` 调用 `wx-manager-content`
- `outfitManage/outfitEdit` 调用 `wx-manager-content`
- `pushSend` 调用 `wx-manager-wework`
- 管理端鉴权：管理员身份校验从 JWT Token 改为 `cloud.getWXContext()` + 数据库角色判断

### 7.2 管理后台改造

**改动范围：大**。核心变化：

1. **安装 CloudBase JS SDK**：`npm install @cloudbase/js-sdk`
2. **初始化 CloudBase**：
   ```javascript
   import cloudbase from '@cloudbase/js-sdk'
   const app = cloudbase.init({
     env: 'your-env-id',
     accessKey: 'your-publishable-key',
   })
   ```
3. **认证替换**：Shiro + Session → CloudBase Auth（用户名/密码）
4. **API 调用替换**：`axios`（27 个 API 模块文件） → `app.callFunction()`
5. **文件上传替换**：本地上传 → `app.uploadFile()`
6. **路由改为 Hash 模式**：适配静态托管 SPA
7. **构建配置**：`publicPath` 改为相对路径

### 7.3 H5 移动端

**暂不迁移**，保留当前 REST API 架构。原因：
- H5 端独立于微信生态，不能使用 `wx.cloud`
- 访问量极低，优先级最低
- 后续如需迁移，可走 CloudBase HTTP 触发云函数

## 8. 认证迁移详解

### 8.1 小程序端

```
当前流程：
  wx.login() → code → POST /wx/auth/login_by_weixin → Java 获取 openid → 返回 JWT Token
  app.js 中 silentLogin() 自动静默登录，501 自动重试

迁移后流程：
  wx.cloud.callFunction() → 云函数通过 cloud.getWXContext() 直接拿到 OPENID → 无需手动登录

  首次使用：OPENID → 查询/创建 litemall_user → 返回用户信息
  后续使用：OPENID → 直接查询 litemall_user
```

### 8.2 管理后台

```
当前流程：
  验证码(kaptcha) → 用户名+密码 → Shiro Session → Cookie
  前端 js-cookie 存储 X-Litemall-Admin-Token

迁移后流程：
  CloudBase Auth.signInWithPassword({ username, password })
  → 获得 access_token → JS SDK 自动携带到后续 callFunction
  → 云函数端通过 auth.getUserInfo() 获取调用者身份 → 权限校验
```

### 8.3 Admin 权限体系保留

现有的 RBAC（用户 → 角色 → 权限）模型在云函数端保留：
- `admin-auth` 云函数内实现权限校验中间件
- 其他 Admin 云函数在入口统一鉴权
- 数据库存 `litemall_role`、`litemall_permission` 表结构不变

### 8.4 管理端权限（小程序管理端）

```
当前流程：
  JWT Token + WxManagerXxxController 中的 hasPermissions() 校验

迁移后流程：
  wx-manager-* 云函数入口统一鉴权：
  cloud.getWXContext().OPENID → 查 litemall_user 的 role → 判断是否为管理员
```

## 10. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| **~360 个 API 重写工作量大** | 整体工期长 | 分阶段交付，WX 商城端优先 |
| **WX 管理端 46 个接口是新模块** | 方案中未预估 | 独立为 wx-manager-* 云函数组，与商城端并行开发 |
| **云函数冷启动** | 首次请求慢（1-3 秒） | 预置并发 + 定时预热 + 函数分组减少数量 |
| **MySQL 连接数管理** | 云函数多实例可能耗尽连接数 | 共享连接池层 + connectionLimit=5 |
| **Java → Node.js 业务逻辑遗漏** | 功能回归 | 逐 Controller 对照，写测试用例 |
| **DelayQueue → 轮询方案** | 订单取消时效从精确延迟变为轮询间隔 | 接受最大 5 分钟延迟（电商场景可接受） |
| **OSS → COS 文件迁移** | 需批量迁移现有图片到云存储 | 编写迁移脚本（OSS 下载 → 云存储上传 → 更新数据库 URL） |
| **管理后台静态托管域名** | 平台默认域名有频率限制 | 后台访问量极低，默认域名完全够用 |
| **H5 端兼容** | H5 仍用 REST API，需后端保留或用 HTTP 触发 | H5 暂不迁移，或用 CloudBase HTTP 触发云函数 |
