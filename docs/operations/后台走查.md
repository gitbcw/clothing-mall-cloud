## 三、会员足迹模块

### 3.1 表格字段问题

| 问题 | 当前状态 | 问题说明 | 优先级 |
|------|----------|----------|--------|
| 用户ID无名称 | 只显示用户ID | 运营人员无法知道是哪个会员 | P1 |
| 商品ID无名称 | 只显示商品ID | 运营人员无法知道浏览了哪个商品 | P1 |

**问题分析**：
- 用户ID（如 `123456`）对人眼无意义，运营人员看不出这是谁
- 商品ID（如 `789`）同样无法让人知道具体是哪个商品
- 需要同时显示「可读名称」便于人工识别

### 3.2 优化建议

| 字段 | 当前 | 建议 | 说明 |
|------|------|------|------|
| 会员标识 | 用户ID | **用户ID + 会员名称** | ID用于定位，名称用于识别 |
| 商品标识 | 商品ID | **商品ID + 商品名称** | ID用于定位，名称用于识别 |
| 足迹ID | 显示 | **隐藏** | 纯技术主键，无管理价值 |

**优化后表格示意**：

```
┌──────────┬────────────┬──────────┬────────────────┬──────────────┐
│ 会员ID   │ 会员名称   │ 商品ID   │ 商品名称       │ 浏览时间     │
├──────────┼────────────┼──────────┼────────────────┼──────────────┤
│ 10001    │ 张三       │ 20056    │ 春季新款外套   │ 03-19 14:30  │
│ 10002    │ 李四       │ 20078    │ 夏季短袖T恤   │ 03-19 15:20  │
└──────────┴────────────┴──────────┴────────────────┴──────────────┘
```

## 四、意见反馈模块（正面案例）

### 4.1 设计亮点

| 字段 | 显示 | 价值 |
|------|------|------|
| 用户名 | ✓ | 运营人员能看懂是谁 |
| 用户ID | ✓ | 可锁定具体用户 |
| 手机号 | ✓ | 可联系用户 |
| 反馈ID | ❌ 或隐藏 | 无管理价值 |

**评价**：此页面设计合理，运营人员能快速理解信息：
1. 看用户名知道是谁
2. 有用户ID可精准定位
3. 有手机号可联系跟进

### 4.2 可作为参考模板

意见反馈页面的字段设计可作为其他页面的参考标准：
- **可读字段优先**：用户名 > 用户ID
- **联系信息保留**：手机号便于运营跟进
- **冗余ID隐藏**：反馈ID等纯技术标识隐藏

## 八、通用设计原则

### 8.1 表格字段设计规范

**核心原则**：只显示对管理有价值的信息

| 判断标准 | 显示 | 隐藏 |
|----------|------|------|
| 信息价值 | 管理员需要看/操作 | 仅系统内部使用 |
| 可读性 | 人眼可理解 | 技术标识符 |
| 决策相关 | 影响管理决策 | 无关决策 |

**ID类字段显示规则**：

| 场景 | 是否显示ID |
|------|-----------|
| 唯一标识需要人工核对时 | ✓ 显示（如订单号） |
| 纯技术主键 | ❌ 隐藏 |
| 关联查询需要时 | ❌ 隐藏（通过名称/标题代替） |

### 8.2 适用页面清单

以下页面需按上述原则审查表格字段：

| 页面 | 待审查字段 | 预期操作 |
|------|-----------|----------|
| 收货地址 | 地址ID、用户ID | 隐藏 |
| 收藏列表 | 收藏ID | 隐藏 |
| 搜索历史 | 搜索ID | 隐藏 |
| 足迹列表 | 足迹ID | 隐藏，增加会员名称+商品名称 |
| 其他列表页 | 按原则审查 | 按需调整 |
