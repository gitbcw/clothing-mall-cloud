## 一、总体结论

本地后台的大部分核心页面可以打开，订单列表、售后列表、商品列表、分类、场景、穿搭、节日商品、平台配置、统计页的首屏渲染整体可达。

但当前存在 5 个明显问题，其中 2 个属于高风险安全漏洞，2 个会直接影响后台正常使用，1 个会影响运营识别数据：

| 优先级 | 问题 | 影响 |
|--------|------|------|
| P0 | 管理员 Token 可伪造 | 任意人只要知道管理员 ID 就能伪装登录后台 |

## 二、问题明细

### 2.1 P0 - 管理员 Token 可被客户端直接伪造

**现象**：
- 后台管理员 token 不是签名 token，也没有服务端会话存储。
- 只要拼出 `admin-{管理员ID}-{当前时间戳}` 这种格式，并且该管理员未删除、时间在 7 天内，就会被后端当成已登录管理员。
- 本次本地巡检就是通过伪造 token 进入后台并访问各路由，说明该问题可以被实际利用。

**影响**：
- 任意知道管理员 ID 的人都可以绕过登录直接访问后台。
- 旧 token 无法主动失效，登出仅清前端状态，无法真正撤销会话。
- 如果管理员密码泄露或浏览器 token 被拿到，风险窗口至少持续 7 天。

**代码位置**：
- `cloudfunctions/layers/layer-auth/nodejs/lib/admin-auth.js:62`
- `cloudfunctions/layers/layer-auth/nodejs/lib/admin-auth.js:86`

**原因说明**：
- `login()` 直接返回 `admin-${admin.id}-${Date.now()}`。
- `adminAuthMiddleware()` 只做字符串拆分、时间校验和管理员存在校验，没有签名、没有随机性、没有服务端会话比对。

**建议**：
- 改为服务端可校验的签名 token 或服务端 session。
- 登出、改密、角色变更后应能立即使旧 token 失效。

### 2.2 P0 - 默认管理员账号密码被公开且登录页默认填充

**现象**：
- 登录页默认填充 `admin123 / admin123`。
- 测试文档也直接写明管理后台账号为 `admin123 / admin123`。

**影响**：
- 后台登录入口对弱口令完全暴露。
- 即使没有上面的 token 伪造漏洞，这也是一个可直接撞库/试探的高风险入口。
- 对正式环境和测试环境都会造成误导，容易让默认口令长期留存。

**代码位置**：
- `clothing-mall-admin/src/views/login/index.vue:60`
- `docs/test/test-plan.md:16`
- `docs/test/test-plan.md:92`

**建议**：
- 去掉前端默认填充账号密码。
- 清理文档里的真实可用口令，改成临时凭据或独立测试账号。
- 强制初始化改密，并增加失败次数限制。

### 2.3 P1 - 登录页与 Dashboard 首屏存在 CloudBase 调用时序问题

**现象**：
- 本地直接打开登录页时，页面会出现 `云函数调用失败: Cannot read properties of null (reading 'scope')`。
- 路由巡检时，`/#/dashboard` 首屏也捕获到了同类错误，集中出现在 `admin-auth/nnotice`、`admin-stat/statGrowth` 等初始化请求上。

**影响**：
- 冷启动时登录页就可能先报错，影响登录信心。
- 登录后首页通知、统计卡片存在首屏加载失败风险，表现不稳定。
- 这类问题会放大“偶发打不开、刷新又好了”的运营感知问题。

**代码位置**：
- `clothing-mall-admin/src/utils/cloudbase.js:15`
- `clothing-mall-admin/src/utils/cloudbase.js:23`
- `clothing-mall-admin/src/utils/request.js:327`
- `clothing-mall-admin/src/views/login/index.vue:86`
- `clothing-mall-admin/src/views/login/index.vue:94`

**原因说明**：
- `cloudbase.js` 里虽然定义了 `ensureAuth()`，但只是启动时异步触发，没有被 `request()` 显式等待。
- 登录页 `created()` 阶段又会立刻调用 `getKaptcha()`，即使验证码已经禁用，也会触发一次云函数请求。
- 结果就是 SDK 认证状态还没准备好时，页面初始化请求已经发出，首屏直接报错。

**建议**：
- 在请求层统一等待 CloudBase 初始化完成后再 `callFunction`。
- 验证码已禁用时，登录页不应继续请求 `auth/kaptcha`。

### 2.4 P1 - 系统设置-管理员页面会触发运行时异常

**现象**：
- 巡检 `/#/system/admin` 时捕获到前端渲染错误：`Cannot read properties of null (reading 'length')`。
- 问题发生在管理员角色展示阶段，页面在角色选项尚未返回时就开始渲染列表。

**影响**：
- 后台管理员账号管理页不稳定，影响新增/编辑管理员。
- 一旦接口慢、接口失败或首屏并发较高，就容易出现白屏/局部报错。

**代码位置**：
- `clothing-mall-admin/src/views/sys/admin.vue:123`
- `clothing-mall-admin/src/views/sys/admin.vue:163`

**原因说明**：
- `roleOptions` 初始值是 `null`。
- `formatRole()` 里直接访问 `this.roleOptions.length`，没有空值保护。
- `getList()` 与 `roleOptions()` 是并行请求，管理员列表先回来时就会触发渲染异常。

**建议**：
- `roleOptions` 默认值改为空数组。
- `formatRole()` 增加空值保护。

### 2.5 P1 - 会员行为页商品图片失效，运营无法快速识别商品

**现象**：
- 在 `/#/platform/collect`、`/#/platform/footprint`、`/#/platform/history`、`/#/platform/address` 巡检中，商品图请求出现 `net::ERR_BLOCKED_BY_ORB`。
- 进一步对失败图片地址做 `curl -I` 检查，返回的是 `404` 且 `content-disposition: attachment`，不是正常可展示图片。

**影响**：
- 会员收藏、足迹、历史等页面失去“看图识货”的能力。
- 运营只能靠 ID/文本识别商品，排查行为数据效率明显下降。

**代码位置**：
- `clothing-mall-admin/src/main.js:76`
- `clothing-mall-admin/src/main.js:80`
- `clothing-mall-admin/src/views/user/collect.vue:40`
- `clothing-mall-admin/src/views/user/footprint.vue:45`

**原因说明**：
- 全局 `imageUrl()` 直接把相对路径拼到固定 COS 域名上。
- 当前部分历史图片对象已不存在或返回头不适合浏览器直接展示，导致浏览器拦截。

**建议**：
- 图片应通过真实可访问 URL 或临时链接展示，而不是无校验地拼固定基址。
- 会员行为页需要兜底占位图，避免整列空白。

## 四、建议处理顺序

1. 先修复管理员认证：Token 签名 / 会话失效 / 默认口令清理。
2. 再处理登录首屏初始化：CloudBase 鉴权时序 + 去掉无效验证码请求。
3. 然后修复系统管理员页空指针，避免后台基础运维功能不可用。
4. 最后补会员行为页图片链路和占位兜底，提升运营可读性。
